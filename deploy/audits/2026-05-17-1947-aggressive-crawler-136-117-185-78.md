# Audit: aggressive crawler at 136.117.185.78 — fake mobile UA, 70 reqs/5 min, all clustered at ~3.8 s

- **Detected:** 2026-05-17 19:47 local
- **Severity:** low-medium — adds load but not yet abusive
- **Source:** caddy access log

## Evidence

70 requests in 5 minutes from `136.117.185.78`. UA claims to be a Pixel 6 / Chrome 114 — but Chrome 114 is two years old and the request rate doesn't look human:

Every response is in the 3.67–3.80 s band, suggesting requests are being throttled by something (Cloudflare? Next.js render cost?) but not blocked. The client also generated 27 of the 27 aborted (`status:0, size:0`) responses in the window — it gave up before the body finished.

## Hypothesis

Most likely an SEO / price-monitoring scraper. The combination is suspicious: fake stable mobile UA, no `Referer`, hits sitemap-derived paths (`/c/*`, `/product/*`, `/search?brand=*`, `/feed.xml`), bursty but not flooding. The clustered 3.8 s response time is the SSR cost — every page is a fresh render because (a) no `Cache-Control` headers from Next.js, (b) per-product pages are unique. So each crawler hit pays full render cost, which directly worsens the image-optimizer load (the SSR triggers `<Image>` warmups for product cards).

## Fix steps

### If we want to keep the bot
- Add `Cache-Control: public, s-maxage=300, stale-while-revalidate=3600` (or similar) on `/c/*`, `/product/*`, `/feed.xml` so Cloudflare serves repeat hits without touching origin. SSR cost goes to zero for them after the first request.

### If we want to block it
- Cloudflare → WAF → Custom Rule: block IPs that send `>30 req/min` to `/c/*` `/product/*` `/search*` from a non-search-engine UA. This catches anything sketchy without blocking real users.
- For known bad: add `136.117.185.78` to a Cloudflare IP block list. (Operator action — not doable from this session.)
- For the moment, `fail2ban` only watches `sshd`. We are not blocking HTTP abuse anywhere on the origin. fail2ban stats: 14 303 failed SSH attempts, 2 342 historical bans — so the abuse surface is real but currently only mitigated at the SSH layer.

## Similar issues to scan for

- Look at the rolling top-N client IPs across longer windows (1 hour, 24 hours). This snapshot is 5 min; the real ranking may be very different.
- Confirm: are real human page views also taking 3.8 s? If yes, the slow path isn't bot-induced contention — it's a baseline SSR cost. Probe with `scripts/probe-cf.mjs` (see CLAUDE.md) and compare against synthetic load.
- Check whether `/mcp` (3 requests in this window) is being probed by an exploit-scanner — that path is unusual.
- No HTTP-layer rate limiting in Caddy. Worth wiring in `caddy-ratelimit` or doing it in front via Cloudflare.
