[30] BUG — feed.xml top-level <updated> lags ingestion by ~14h (FIXED)
   • Probe at 10:58 UTC: GET https://teno-store.com/feed.xml returned
     <updated>2026-05-11T18:09:18.419Z</updated> — exactly 14 hours
     before the request. Meanwhile catalog.products has rows ingested
     as recently as 2026-05-12T08:37:18Z (minutes before the probe).
   • Root cause: hits are sorted by source postedAt (the seller's
     original posting date — what buyers want for "newest"). The feed
     used hits[0].updatedAt as the feed-level Atom <updated>. But when
     today's ingestions all have source postedAt OLDER than yesterday's
     hits[0] postedAt, hits[0].updatedAt doesn't move, so the feed's
     <updated>, ETag, and Last-Modified all stay frozen. Conditional
     GET (If-None-Match / If-Modified-Since) returns 304 indefinitely
     and feed readers stop seeing the new listings.
   • FIX (applied): take max(updatedAt) across all hits instead of
     reading hits[0]. Same ordering, same entries — only the feed-
     level freshness signal changes. Per Atom RFC 4287, <updated>
     should be "the most recent instant the feed was modified", which
     this matches; hits[0].updatedAt was the previous (incorrect)
     interpretation.

[28] BUG — Checkout cart-empty-after-201 root cause likely identified

   • Confirmed Caddyfile only retries on dial errors, NOT on a 200/201

   • Refined theory: an MCP/agent client retries POST /v1/checkout/

     genReqId honors X-Request-Id if present, which is why the
     "Promise errored, but reply.sent" line shares the same reqId as
     the successful 201 — they are TWO physical request attempts
     unified into one Pino log thread by the client-supplied id.
   • Evidence: 5 errors in 1h, 4 of which were POST /mcp (the MCP

     deterministic-retry semantics here.

       a) Add OrderRepo.findRecentByCartId(cartId, withinMs=600_000)
       b) In the route, when c.lines.length === 0, look up a recent
          order for this cartId. If one exists with a matching
          customer.phone (or no customer mismatch), return it as if
          it were freshly created. Otherwise fall through to the
          existing 400-validation throw.
       c) Same pattern in packages/mcp-server/src/tools/buyer.ts
          checkout.confirm handler.

     /v1/checkout/confirm itself clears the cart on success (line 118

     prior order body is the spec-correct response.

     ValidationError into a 409 with a clear "order_already_confirmed"
